### 0.7 内容生成类任务处理规范（2026-02-21 新增）

当执行**内容生成类任务**（如文字优化、文案创作、思维模型字段优化等）时，必须遵循以下规范：

#### 核心原则

- **禁止使用批量脚本替换**：内容生成类任务必须由AI生成具体内容，不能使用简单的批量脚本替换
- **分批处理**：每批处理5个模型，用AI生成具体内容
- **全字段优化**：如果一个模型需要优化，则该模型的所有套话字段都需要优化

#### 处理流程

```
识别套话 → 分批处理（每批5个）→ AI生成具体内容 → 验证JSON → 继续下一批
```

#### 套话识别模式

| 字段 | 套话模式 | 正确模式 |
|------|----------|----------|
| deepAnalysis | "这是一个思维模型，展示了XXX的核心概念和应用。" | 解释模型的本质、来源、核心洞察 |
| principle | "思维模型的核心原则：XXX" | 3-5条具体原则，每条换行 |
| scope | "思维模式、决策分析、问题解决" | 具体应用领域，用顿号分隔 |
| tips | "1. 理解模型的核心概念；2. 结合实际问题应用；3. 定期复盘和优化" | 3-5条具体可操作的技巧，每条换行 |
| practice | "将XXX应用到日常决策和问题解决中，观察效果并调整。" | 3-5条具体实践步骤，每条换行 |

#### 全字段优化原则

**重要规则**：当一个模型需要优化时，该模型的**所有套话字段都需要优化**，不能只优化一个字段。

例如：如果发现某个模型的 `practice` 字段是套话，那么必须检查该模型的所有其他字段（deepAnalysis、principle、scope、tips），如果其他字段也是套话，必须全部优化。

#### 任务管理

- 使用TodoWrite工具创建分批任务列表
- 每批任务完成后标记为completed
- 持续处理直到所有任务完成

## 1. 用户工作流程偏好

### 1.1 智能体自主工作模式
- **核心原则**：AI应主动完成整个工作闭环，不等待进一步指令
- **工作流程**：发现问题 → 自动检查错误日志 → 提出修复方案 → 执行修复 → 验证修复 → 交付成果
- **用户期望**：当代码出错时，AI能智能体自动主动检查错误日志并提出修复方案，而非等待进一步指令

### 1.2 界面验证要求
- 修复后必须打开预览验证效果
- 逐个点击所有功能模块测试是否有问题
- 挨个检查，有问题的修复
- 修复后再检查，直到没有问题为止
- **全权委托原则**：用户希望AI直接打开预览，不需要用户手动点击预览按钮

### 1.3 问题修复标准
- 完成标志：明确告知"成功了，能打开了，没错误了"
- 避免重复修复：如果修复了很多次仍有问题，需要从架构层面思考根本原因
- 主动识别架构问题：当单个模块过大或问题频发时，主动提出重构建议

## 3. 项目主题与UI规范

### 3.1 拟态深色主题标准
- **标准背景色**：`#1e1e2e`（不是#2a2d36）
- **颜色一致性要求**：需要全局搜索并统一所有拟态深色模式的背景色
- **检查范围**：包括所有组件、配置文件、CSS样式

### 3.2 深色模式适配规范
- **背景层级**：主背景、卡片背景及所有嵌套容器在深色模式下必须应用深色主题
- **禁止白色残留**：不允许在深色模式下出现白色背景残留
- **文字对比度**：所有文本元素（特别是按钮文字）在深色背景下需保证足够对比度
- **全面覆盖**：包括荣誉殿堂、思维中心、作战中心等所有界面

### 3.3 主题切换配置
- **默认主题**：拟态浅色
- **切换顺序**：拟态浅色 → 拟态深色 → 普通浅色 → 普通深色（按此循环）

### 3.4 UI设计规范

#### 帮助按钮规范
- **尺寸标准**：全局小问号图标使用14px尺寸（`size={14}`）
- **样式规范**：
  - 使用`transition-colors`实现颜色过渡效果
  - 外层容器使用`hover:scale-[1.1]`实现缩放效果
  - 配合`transition-all duration-300`实现平滑过渡

#### 拟态风格按钮
- **设计风格**：Neomorphism（拟态设计）
- **SVG图标**：优先使用SVG图标而非图片
- **一致性要求**：所有模块按钮保持统一的拟态风格

#### 深色模式文本可读性
- 确保在深色背景下文本颜色有足够对比度
- 特别关注按钮文字、标签文字的可见性
- 避免低对比度导致的阅读困难

## 4. 前端开发规范

### 4.1 组件开发规范
- **代码行数限制**：单个组件不超过500行
- **单一职责原则**：每个组件只负责一个功能模块
- **组件复用原则**：优先使用组件复用和模块继承
- **逻辑与UI分离**：使用自定义Hook管理复杂逻辑

### 4.2 代码结构要求
- **避免过度嵌套**：最多3层嵌套
- **TypeScript严格模式**：使用TypeScript严格类型检查
- **避免any类型**：保持类型安全

### 4.3 HTML规范
- **禁止按钮嵌套**：button元素不能嵌套（会导致Hydration错误）
- **正确的事件处理**：直接将onClick传递给自定义组件，而不是包裹额外的button

### 4.4 UI样式管理
- **音频列表排序**：按播放次数降序排列
- **Select样式**：深色模式下select组件需要特殊样式适配
- **待完成任务排序**：优先显示高优先级任务

### 11.3 JSON文件编辑规范（2026-02-21 新增）

- **禁止全局CSS重置**：visualDesign字段中禁止使用 `* { margin: 0; padding: 0; }` 等全局选择器
- **使用类选择器**：所有CSS样式必须使用 `.neu-card` 等类选择器，避免影响全局布局
- **编辑前备份**：编辑大型JSON文件前，先用git stash或复制备份
- **验证JSON有效性**：编辑后必须验证JSON语法正确性
- **SearchReplace谨慎使用**：对于包含复杂HTML/CSS的字符串字段，避免使用SearchReplace进行部分替换
- **换行必须使用 `\n`**：JSON字符串中不能直接换行，必须使用 `\n` 转义字符
  - ❌ 错误：`"text": "第一行\n第二行"`（直接换行会导致JSON解析错误）
  - ✅ 正确：`"text": "第一行\n第二行"`
- **编辑后验证**：使用 `node -e "JSON.parse(require('fs').readFileSync('file.json'))"` 验证JSON有效性

### 11.4 错误修复机制（2026-02-21 新增）

当 AI 犯错被用户修正后，应按照以下流程处理：

#### 修复流程

```
发现错误 → 定位原因 → 执行修复 → 转化为规范 → 更新文档 → 验证效果
```

#### 规范转化规则

| 错误类型 | 转化位置 | 示例 |
|----------|----------|------|
| JSON编辑错误 | `project_rules.md` 11.3 JSON文件编辑规范 | 换行必须用 `\n` |
| UI样式错误 | `project_rules.md` 第3章 UI规范 | 深色模式背景色 |
| 组件开发错误 | `project_rules.md` 第4章 开发规范 | 禁止按钮嵌套 |
| 功能模块错误 | `docs/` 对应规范文档 | 思维模型套话识别 |

#### 已转化的规范

| 原错误 | 转化后的规范 | 规范位置 |
|--------|--------------|----------|
| JSON字符串直接换行 | JSON换行必须使用 `\n` 转义字符 | 11.3 JSON文件编辑规范 |
